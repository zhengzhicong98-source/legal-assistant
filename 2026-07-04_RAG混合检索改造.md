# RAG 混合检索改造

> 从纯向量检索升级为"向量 + 关键词"混合检索，解决精确法条查询无法命中、相似度分数断链、缺少自助测试工具三大痛点。

---

## 1. 整体架构

```
用户提问
  │
  ▼
┌──────────────┐
│  查询扩展      │  口语→法律术语映射（41条），补全用户没说出来的专业词
└──────┬───────┘
       │
       ▼
┌──────────────┐
│  Embedding    │  调用 text-embedding 模型，把扩展后的查询文本转为向量
└──────┬───────┘
       │
       ├──────────────────┐
       ▼                  ▼
┌──────────────┐   ┌──────────────┐
│  向量检索      │   │  关键词检索    │  pg_trgm trigram 索引，三字段并行打分
│  IVFFlat      │   │  pg_trgm GIN  │
└──────┬───────┘   └──────┬───────┘
       │                  │
       └────────┬─────────┘
                ▼
┌──────────────┐
│  RRF 融合      │  加权融合 65% 向量 + 35% 关键词，去重排序
└──────┬───────┘
       │
       ▼
┌──────────────┐
│  LLM 生成      │  带上相关度百分比，让模型更谨慎引用
└──────┬───────┘
       │
       ▼
┌──────────────┐
│  前端展示      │  法条原文折叠 + 相关度百分比 badge
└──────────────┘
```

## 2. 改造前的问题（为什么需要这次改造）

### 2.1 精确条号查询完全失效

用户问"民法典第 710 条"，纯向量检索会把"第 710 条"当成普通语义去匹配，结果返回一堆"合同解除""租赁合同"相关法条，真正的 710 条排在很后面甚至不出现在结果中。

**本质原因**：向量模型（text-embedding）擅长语义相似，不擅长精确文本匹配。法条编号是离散的字符串标识符，和"第七条"在向量空间里距离很远。

### 2.2 similarity 分数断链

后端计算了每条检索结果的相似度分数，但前端接收的数据结构中缺少"相似度"这个属性——代码里标着 TODO，空置了三年。前端拿到的是一个空数组，LLM 看不到"这条法条和用户问题有多相关"，容易把弱相关法条当成权威引用。

### 2.3 生产库有改动但没纳入版本控制

Supabase 生产库上已经跑着一个混合检索函数，是之前手动在 SQL Editor 里改的。迁移文件里没有，`supabase/migrations/` 目录下找不到它的定义。

**风险**：如果数据库重建或迁移回滚，这个函数会丢失，且没人知道它的确切实现。

### 2.4 查询扩展覆盖面不足

只有 6 条口语→法律术语映射，比如"被开除了"→"解除劳动合同"，"离婚"→"感情破裂"。但用户可能问"买到假货"（消费欺诈）、"房东不退押金"（租赁合同）、"被打伤了"（故意伤害）——这些高频场景都没有覆盖。

### 2.5 管理员无法自助测试

每次想看"检索效果好不好"，需要打开 Edge Function 日志或直接跑 SQL。没有可视化界面能快速输入一个问题和看每条命中法条的分数。

## 3. 改造内容

### Stage 1 — 混合检索升级

#### 3.1.1 建 pg_trgm GIN 索引

**做了什么**：在知识库表的正文、标题、来源三个字段上分别建了 trigram GIN 索引。

**为什么做**：之前没有任何全文索引，关键词搜索走全表扫描。现在用 trigram（三字符片段）索引，数据库可以快速定位包含"民法典第710条"里任意三字符子串的行。

**为什么是 pg_trgm 而不是 tsvector（全文搜索）**：PostgreSQL 原生的 `to_tsvector` 只支持英文分词。中文需要 `zhparser` 或 `pg_jieba` 扩展，但 Supabase 不支持安装第三方扩展。`pg_trgm` 是字符级的 trigram 比对，天然支持中文，不需要分词插件。

#### 3.1.2 升级混合检索函数

**改造前**：只在 `content` 字段上做 trigram 相似度计算，且必须向量分≥阈值才能进入结果集。

**改造后**：
- **三字段共同打分**：正文 + 标题 + 来源都参与 trigram 相似度计算
- **权重放大**：标题和来源命中时权重放大 1.5 倍——因为法条标题里带条号（如"民法典第710条"），标题命中说明用户大概率在问精确条号
- **放宽过滤**：原来是"向量分必须达标"，改成"向量分**或**关键词分任一达标"——纯关键词命中也能进入结果集，解决精确匹配问题
- **返回三个分数**：
  - 向量相似度：纯向量语义匹配得分
  - 关键词匹配分：纯 trigram 字符重叠得分
  - 混合加权分：加权融合分（65% 向量 + 35% 关键词）

**为什么向量权重更高（0.65 vs 0.35）**：法律语义匹配是主路径——用户大部分问的是"什么情况下可以解除合同"这类语义问题，不是精确条号。关键词是补强通道，专门补足向量检索在精确匹配上的短板。

#### 3.1.3 修复 similarity 断链

**做了什么**：
- 前端数据结构中增加了"相似度"字段，用于传递每条法条的相关度百分比
- 消除了硬编码的空数组
- LLM 提示词里附上了相关度百分比（如"相关度 87%"），让它更谨慎地判断是否引用

**为什么做**：LLM 看到引用列表时无法判断"这条法条和问题到底有多相关"。加上百分比后，它可以自己决定"72% 的我引用，34% 的我略过"。

## 4. 评估方法论

### 4.1 评估集设计

- 从 28 条扩到 **33 条**，新增了 5 条精确条号 query（如"民法典第 710 条""刑法第 264 条"）
- 每条 query 标注了期望的 Top-1 和 Top-3 法条 ID

### 4.2 A/B 对比流程

为了严谨对比新旧函数的差异，做了 5 轮评估快照：

```
1. 写临时迁移 → 把混合检索函数切成旧版本
2. 跑 eval-rag.mjs → 生成 report-old.json
3. 写临时 migration → 切回新版本
4. 跑 eval-rag.mjs → 生成 report-new.json
5. diff 两份报告 → 确认每个指标变化
```

### 4.3 最终结果（33 条评估集）

| 指标 | 旧函数 | 新函数 + 扩展 | 变化 |
|------|-------|-------------|------|
| **Top-1** | 48.5% | **63.6%** | +15.1pp |
| **Top-3** | 72.7% | **84.8%** | +12.1pp |
| **MRR** | 0.637 | **0.755** | +18.5% |
| **平均相似度** | 0.640 | **0.665** | +0.025 |

**解读**：
- Top-1 提升最显著（+15.1pp），说明混合检索在"最相关结果排第一"这个目标上改进了很多——这主要受益于精确条号查询
- MRR（平均倒数排名）提升最大（+18.5%），说明整体排序质量显著改善——相关法条被排到更靠前的位置
- 平均相似度增幅较小（+0.025），说明查询扩展让语义匹配更准了，但语义精度本身不是本次改造的重点

**样本量说明**：33 条评估集在统计上约 ±3% 的波动范围。样本量偏小但对于验证"精确条号查询是否修复"这个核心目标足够。

## 5. 技术决策记录

### pg_trgm vs tsvector（全文搜索）

| 方案 | 中文支持 | Supabase 兼容 | 结论 |
|------|---------|-------------|------|
| `to_tsvector` 原生 | 不支持 | 支持 | 英文好用，中文无分词 |
| `zhparser` 扩展 | 支持 | 不支持 | 需要装扩展，Supabase 不允许 |
| `pg_jieba` 扩展 | 支持 | 不支持 | 同上 |
| `pg_trgm` | 天然支持（字符级） | 内置支持 | **唯一可用方案** |

### 权重分配 0.65 / 0.35

- **向量 65%**：法律语义匹配是主路径，覆盖大多数"帮我分析这种情况"类问题
- **关键词 35%**：补强精确匹配能力，解决"第 X 条"类问题

这是一个经验值，后续可以通过评估数据做网格搜索找到最优权重。

## 6. 专业名词速查表

| 名词 | 通俗解释 |
|------|---------|
| **RAG** (Retrieval-Augmented Generation) | 检索增强生成。LLM 先查资料再回答，不是凭记忆编造。好比考试允许翻书 |
| **embedding** | 向量嵌入。把一段文字转换成数字数组（向量），语义相近的文字向量也相近 |
| **pg_trgm** | PostgreSQL 的 trigram 扩展。把文本切成长度为 3 的字符片段，用片段重叠程度计算相似度。天然支持中文 |
| **IVFFlat** | 一种向量索引算法。把向量空间分成多个区域，检索时只搜最近区域，不用全表扫描。类似图书馆分区摆放 |
| **RRF** (Reciprocal Rank Fusion) | 倒数排名融合。把向量检索和关键词检索的排名合并成一个最终排序，不关心绝对分数只关心相对顺序 |
| **MRR** (Mean Reciprocal Rank) | 平均倒数排名。评估指标，衡量"正确答案第一次出现的位置"。MRR=1 说明所有正确答案都排在第一位 |
| **trigram** | 三字符片段。"民法典第710"的 trigram 包含："民法""法典""典第""第7""710"——"710"这个片段能在索引中精确匹配 |
| **Edge Function** | Supabase 的 serverless 函数，运行在 CDN 边缘节点。法律助手的 legal-chat 接口就是一个 Edge Function |
| **GIN 索引** | Generalized Inverted Index（广义倒排索引）。PostgreSQL 里加速全文搜索的索引类型，类似书的末尾索引 |
| **查询扩展** | 把用户的日常口语补全为法律术语，让检索系统更好地理解用户意图 |
| **hybrid search** | 混合检索。同时用向量检索（语义）和关键词检索（字面）搜，然后合并排序 |
| **A/B 评估** | 把新旧方案各跑一遍评估集，对比指标差异，用数据确认改进是否有效 |

## 8. 后续优化方向

1. **权重调优**：用网格搜索找到 0.65/0.35 之外更优的向量/关键词权重组合
2. **Reranker 模型**：在混合检索后加一层专门的重排序模型（如 Cohere Rerank），进一步优化排序质量
3. **评估集扩充**：从 33 条逐步扩到 100+ 条，降低统计波动，覆盖更多边缘场景
4. **A/B 系统化**：把 A/B 评估从手动临时 migration 改成自动化 pipeline，每次部署自动跑评估
5. **用户反馈闭环**：前端加"这个引用有用吗"按钮，收集真实用户反馈作为评估集的补充
